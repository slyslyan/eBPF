
## 一、eBPF 与内核原理

### Q1：什么是 eBPF？和传统 BPF 有什么区别？
**A**：eBPF（extended BPF）是在传统 BPF（用于包过滤）基础上的扩展，允许在内核中运行沙箱化的程序。主要区别：
- 指令集更丰富，接近通用 RISC
- 支持 Maps 在用户态/内核态之间双向交换数据
- 可通过 kprobe/uprobe/tracepoint/XDP 等挂载到多个内核子系统
- 具备 **验证器**，确保程序不会导致内核崩溃（无死循环、指针越界检查）

### Q2：你的项目中 eBPF 程序挂载到了哪里？为什么选择 `tcp_sendmsg`？
**A**：挂载在 `kprobe/tcp_sendmsg` 和 `kretprobe/tcp_sendmsg`。因为 `tcp_sendmsg` 是 TCP 发送数据时的必经路径，无论是 HTTP、gRPC 还是微服务间调用，最终都会调用该函数。通过入口获取四元组、出口计算延迟，可以无侵入地捕获所有 TCP 连接信息。

### Q3：kprobe 和 tracepoint 有何区别？为何最终选 kprobe？
**A**：
- **kprobe** 可以挂到任意内核函数，但受内核版本、函数符号是否存在影响，且接口不稳定。
- **tracepoint** 是内核暴露的稳定接口，但覆盖的场景有限，提供的数据种类也固定。
- 我们最初尝试 `sys_enter_sendto` tracepoint，发现只适用 UDP；而 `tcp_sendmsg` 只有 kprobe，因此折中选用 kprobe。

### Q4：在内核中读取 `struct sock *sk` 遇到过什么问题？怎么解决的？
**A**：
- 在 `kretprobe` 出口读取 `ctx->di` 获取 sk 时，发现寄存器已被函数执行过程覆盖，读到的是脏数据。
- **解法**：在 `kprobe` 入口就通过 `BPF_CORE_READ_INTO` 将四元组存入 BPF map（按 `pid_tgid` 为 key）。出口仅从 map 读取并计算延迟。

### Q5：什么是 CO-RE？你的项目如何受益？
**A**：CO-RE（Compile Once, Run Everywhere）允许一次编译 eBPF 字节码，在不同内核版本上运行，无需像传统方法那样为每个内核版本编译。我们使用 `vmlinux.h` 和 `BPF_CORE_READ_INTO` 宏，让程序能适配 5.8+ 的各种小版本。

### Q6：为什么在 K8s 容器内无法直接读 `/proc/<pid>/net/tcp` 获取源 IP？
**A**：尽管开了 `hostPID: true`，容器的 Mount Namespace 仍然独立，`/proc` 并不是宿主机完整视图。最终我们通过在内核入口读取 `skc_rcv_saddr`（IPv4）或 `skc_v6_rcv_saddr`（IPv6）解决问题，无需用户态补全。

### Q7：eBPF Ring Buffer 相比 Perf Buffer 有什么优势？
**A**：
- 支持多 CPU 并发写入无需锁，延迟更低。
- 不需要 per-CPU buffer，内存利用率更高。
- 提供 `reserve/commit` 两步操作，对动态大小事件友好。

## 二、网络监控与延迟计算

### Q8：为什么用 `tcp_sendmsg` 得到的延迟是 0.0x ms，而 `curl` 实际等了 2s？
**A**：`tcp_sendmsg` 只负责把用户数据拷贝到内核 socket 缓冲区，拷贝完成即返回，不等待 TCP 握手或网络传输。因此，它反映的是**缓冲区写入延迟**，不是端到端网络延迟。我们通过模拟延迟开关（`SIMULATE_LATENCY`）来验证闭环，真实生产可以换用 `kprobe/tcp_close` 或 `tracepoint/inet_sock_set_state` 获取连接寿命。

### Q9：如何处理 IPv4/IPv6 双栈？Go 程序的套接字是什么类型？
**A**：Go 默认创建 `AF_INET6` 套接字，使用 `::ffff:0.0.0.0/96` 来表示 IPv4 映射地址。此时 `skc_rcv_saddr` 为 0，真实地址藏在 `skc_v6_rcv_saddr.in6_u.u6_addr32[3]` 中。我们在 C 代码中判断 `skc_family`，对 IPv6 提取尾部 4 字节还原 IPv4。

### Q10：如何保证 C 和 Go 结构体的内存对齐一致？
**A**：C 侧显式 padding（`__u16 family; __u8 pad[2]; … __u8 pad2[4]`），使结构体为 48 字节。Go 侧定义 `netEventRaw` 时字段顺序和类型完全对应，并用 `binary.LittleEndian` 解析。

## 三、根因分析算法

### Q11：根因定位算法是怎么设计的？为什么不用简单的阈值比较？
**A**：我们采用了 **EMA（指数移动平均）** + **反向随机游走** 两阶段算法：
1. 每条边维护 EMA 延迟基线，实时计算异常分数（当前平均延迟 / EMA × 错误率因子）。
2. 在异常边的下游节点出发，沿反向调用关系进行带重启的随机游走（Topic-Sensitive PageRank），计算每个节点的嫌疑概率。这种设计可以避免固定阈值的误报，并能处理级联故障（比如 DB 慢导致上游多个服务报警）。

### Q12：什么是反向随机游走？为什么能定位根因？
**A**：它模仿“故障向上游传播”的物理过程。异常首先出现在下游（如数据库变慢），然后引起直接调用者的延迟增加，依次向上。反向随机游走从异常边向下游节点出发，按照边的异常权重向调用者传播概率，经过多次迭代，故障源头节点会累积更高的嫌疑分数。

### Q13：EMA 基线有什么优势？为什么不用固定阈值？
**A**：EMA 能自适应不同服务、不同时段的流量模式。例如，批处理任务在夜间延迟较高是正常现象，固定阈值（如 100ms）会导致大量误报。EMA 通过学习历史行为，只对**偏离正常基线**的边报警，准确率更高。

### Q14：如果整个集群同时变慢（比如机房网络故障），你的算法会如何处理？
**A**：所有边的 EMA 基线会同步上升，比值接近 1.0，不会产生大量异常。这时需要依赖绝对阈值（如我们设置的 200ms）或 **全局指标**（如每个节点的错误率、超时比例）来触发告警。

## 四、自愈与系统设计

### Q15：自愈时如何保证不会误杀集群自身组件？
**A**：我们在 `addDropIP` 函数中维护了一个 `protectedIPs` 集合，包含 `127.0.0.1`、`::1`、Minikube 控制平面 IP `192.168.49.2` 等。限流前先检查，若命中则跳过并打印日志。

### Q16：TC 限流和 K8s Pod 重启如何协同工作？
**A**：
1. 先执行 TC 限流（100% 丢包），立刻在主机层面阻断故障服务流量，防止整个集群被拖垮。
2. 通过 client-go 查找嫌疑 IP 对应的 Pod，执行删除操作（Deployment 会自动重建）。
3. 新 Pod 会获得新 IP，旧的 TC 规则不作用于新 IP，服务恢复。

### Q17：为什么用 `tc netem` 命令而非纯 eBPF TC 程序？
**A**：当前使用的 cilium/ebpf v0.21.0 缺少 `tc` 包，后续升级后可替换。纯 eBPF TC 程序在内核中通过 map 查询直接丢弃包，无需 fork 外部命令，性能更优，更适合生产环境。

### Q18：火焰图抓取的前提是什么？失败如何处理？
**A**：需要目标服务在启动时导入了 `net/http/pprof`，并监听在指定端口（默认 6060）。如果目标没有 pprof，抓取会失败，但不会影响自愈流程，仅记录错误日志。

### Q19：Prometheus 指标如何设计避免高基数问题？
**A**：我们**不暴露原始事件**，只暴露聚合后的边/节点指标。标签只使用服务级标识（如进程名），而非 Pod IP 或实例 ID。Histogram 桶也控制数量（10 个桶），确保时间序列数量可控。

## 五、Kubernetes 与云原生

### Q20：DaemonSet 如何配置才能运行 eBPF 程序？
**A**：必须：
- `hostNetwork: true` （使用宿主机网络栈，看到真实网卡）
- `hostPID: true` （访问宿主机 /proc）
- `privileged: true` 或授予 `CAP_BPF`, `CAP_NET_ADMIN`, `CAP_SYS_PTRACE` 等细粒度权限
- 挂载 `/sys/kernel/debug`

### Q21：为什么在 Pod 内启用 `hostPID` 后仍无法读取宿主机进程的 `/proc`？
**A**：虽然进程列表可见，但容器的 Mount Namespace 独立，`/proc` 挂载点的内容不是宿主机的完整视图。更可靠的做法是在 eBPF 程序中直接获取需要的信息，而非依赖 /proc。

### Q22：如何通过 client-go 根据 Pod IP 找到对应的 Pod？
**A**：使用 `FieldSelector: "status.podIP=<ip>"` 调用 `CoreV1().Pods("").List(...)`，可以获取跨所有命名空间的 Pod。然后通过 Labels 反查 Service、OwnerReference 等信息。

## 六、系统设计与面试开放题

### Q23：如果让你设计一个多集群的统一可观测平台，你会如何扩展当前项目？
**A**：
1. 每个集群部署 DaemonSet，探针通过 gRPC 或 Kafka 将 edge 数据发送到中心处理集群。
2. 中心端做全局拓扑拼接（通过服务名匹配）。
3. 统一 Prometheus，全局 Grafana 展示多集群健康状态。
4. 自愈策略可由中心下发，但限流动作由本地 agent 执行。

### Q24：该项目最大的技术挑战是什么？
**A**：在内核态获取完整的 TCP 四元组并稳定通过验证器。我们经历了 `kprobe` 参数失效、IPv6 双栈、CO-RE 字段读取失败、Ring Buffer 数据错乱等问题，最终通过在入口保存数据、显式 padding、手动处理字节序等方法解决。

### Q25：你怎么评估这个项目的成熟度？离生产环境还有多远？
**A**：
- **已完成**：核心采集、拓扑、根因定位、自愈、火焰图、Prometheus 集成。
- **待完善**：使用 eBPF TC 替代 tc 命令、支持 HTTP/gRPC 协议解析、引入更鲁棒的自适应阈值、多节点聚合、通知系统（飞书/钉钉）。
- **生产就绪度**：约 70%，主要差距在自动化测试和权限优化。

### Q26：如果目标服务没有开启 pprof，如何抓取火焰图？
**A**：我们可以在自愈前通过 eBPF uprobe 挂载到目标进程的 `runtime/pprof` 相关函数，强制触发采样。或者利用 perf event 在内核堆栈跟踪，生成 off-CPU 火焰图。不过这需要更复杂的 eBPF 程序。

### Q27：你的根因分析算法如何处理循环依赖？
**A**：微服务调用在单次请求中是 DAG，但整体拓扑可能存在环（如循环调用）。我们的随机游走通过**迭代固定次数**自然收敛，且概率分布会给循环节点一个稳定值，不会发散。同时，异常边的异常分数限制了传播强度。

### Q28：如何保证 eBPF 程序在不同内核版本间的兼容性？
**A**：使用 CO-RE + BTF。bpftool 生成 `vmlinux.h`，`BPF_CORE_READ_INTO` 自动适配字段偏移。同时我们在 DaemonSet 的 init 容器中检查内核版本和 BTF 支持，不满足则拒绝调度。

### Q29：监控平台自身性能消耗如何控制？
**A**：
- eBPF 程序只读取必要的字段，不分析协议内容。
- Ring Buffer 使用无锁设计，用户态异步消费。
- Prometheus Histogram 自动聚合，不保存原始事件。
- 火焰图只在自愈时触发，而非持续抓取。

### Q30：未来如果要支持更复杂的协议（如 gRPC/WebSocket），你会怎么设计？
**A**：采用 uprobe 挂钩 Go 的 `grpc.Invoke` 或 `net/http.Read`，从参数中提取方法名、路径、状态码等元数据，与 TCP 事件关联。对于加密流量，参考 eCapture 的思想，在 `crypto/tls` 库 `writeRecordLocked` 和 `Read` 函数处挂载 uprobe 获取明文。
